# dddguard



Architecture Guard, Linter, and Scaffolder for Domain-Driven Design projects.



Stop arguing about folder structures and layer boundaries. `dddguard` enforces strict architectural rules, generates compliant boilerplate, and visualizes your project dependency graph.



It is designed to be your interactive architectural companion.



## Quick Start



Install globally using `uv` (requires Python 3.10+):



```bash

uv tool install dddguard

```



Simply run the tool to enter the interactive HUD:



```bash

dddguard

```



The tool will guide you through scanning, linting, or creating new components.

### Anatomy of the Architecture



1.  **`root/`**: The top-level entrypoint for your entire application. Its `composition.py` assembles the different Bounded Contexts and `shared/` components.



2.  **`shared/`**: A place for code that is shared across multiple contexts, such as a shared kernel, common libraries, or cross-cutting concerns like logging. The structure within `shared/` is intentionally flexible.



3.  **`<context_name>/`**: Represents a Bounded Context, a self-contained module with its own internal layers:

    -   **`domain/`**: The heart of the context's business logic. It is completely isolated and pure.

    -   **`app/`**: Orchestrates the domain logic to perform specific tasks, defining the application's capabilities.

    -   **`adapters/`**: Translate between the outside world and the application layer, implementing interfaces defined in `app/` or triggering `app/` logic.

    -   **`dto/`**: Define the data contracts for requests and responses, ensuring clean boundaries.

    -   **`ports/`**: Contain the raw, technology-specific code that connects to infrastructure.

    -   **`composition.py`**: The context's local "glue" that wires together its internal components.

        â””â”€â”€ consumers/

            â””â”€â”€ payment_succeeded_consumer.py

```

## 1. Driven Adapters (Outbound adapters)



These adapters implement interfaces ([[app#5-interface|ports]]) defined in the [[app|app]] layer. They execute application requests to external systems.

### 1.1. Repository



-   **What it is:** Implementation of the mechanism for storing and retrieving [[domain#3-aggregate-root|Aggregate Root]] from persistent storage. Provides the illusion of an in-memory collection.

-   **Location (Heuristic):** Path to module contains `repositories`, `repository`, `repos` or `repo`.

-   **Naming (RegEx):** Module (file) name matches `.*repo(sitory)?$`, `.*provider$`.

-   **Key difference:** Always returns or accepts domain [[domain#2-entity|Entities]]/`Aggregate Roots` and performs mapping between them and the storage data format. Does not perform `commit`.



### 1.2. Transaction Manager



-   **What it is:** Component for managing transaction boundaries (commit/rollback). Implements an interface that allows the `app` layer to control atomicity of changes.

-   **Location (Heuristic):** Path to module contains `transactions`, `transaction`, `uow`, `unit_of_work`.

-   **Naming (RegEx):** Module (file) name matches `.*uow$`, `.*unitofwork$`.

-   **Key difference:** Does not interact with domain objects directly. Its task is to guarantee integrity of changes made within a single business scenario.



### 1.3. Gateway



-   **What it is:** Encapsulates access to external third-party systems (e.g., payment systems, cloud storage, third-party APIs). Transforms our domain/[[dto|DTO]] objects into the external API format.

-   **Location (Heuristic):** Path to module contains `gateways`, `gateway`, `clients`.

-   **Naming (RegEx):** Module (file) name matches `.*gateway$`, `.*client$`.

-   **Key difference:** Works with external APIs that are not part of our system.



### 1.4. ACL Facade



-   **What it is:** Implementation of the Anti-Corruption Layer (ACL) pattern. Protects the domain layer from pollution by models or concepts from another [[01-2_MyProjects - dddguard - guidelines - context_content-summary|Bounded Context]]. Provides an interface to another context.

-   **Location (Heuristic):** Path to module contains `facades` or `acl`.

-   **Naming (RegEx):** Module (file) name matches `.*facade$`, `.*acl$`.

-   **Key difference:** Serves as a "translator" between two Bounded Contexts, smoothing their differences.



### 1.5. Publisher



-   **What it is:** Component responsible for publishing domain or integration events to the outside world (e.g., via message broker). "Fire-and-forget" principle.

-   **Location (Heuristic):** Path to module contains `publishers`, `publisher`, `pub`, `producers` or `broadcasters`.

-   **Naming (RegEx):** Module (file) name matches `.*publisher$`, `.*producer$`, `.*broadcaster$`, `.*sender$`.

-   **Key difference:** Only sends messages, does not await synchronous response.



---



## 2. Driving Adapters (Inbound adapters)



These adapters receive requests from the outside world and initiate execution of [[app#1-use-case|Use Cases]] or [[app#2-query|Queries]] in the application.



### 2.1. Controller



-   **What it is:** Handles inbound synchronous requests (e.g., HTTP, RPC). Responsible for request deserialization, invoking the corresponding `Use Case`/`Query`, and response serialization.

-   **Location (Heuristic):** Path to module contains `controllers`, `controller`, `views`, `view`, `endpoints`, `routers`, `router` or `api`.

-   **Naming (RegEx):** Module (file) name matches `.*controller$`, `.*view$`, `.*router$`, `.*endpoint$`, `.*api$`.

-   **Key difference:** Synchronously processes the request and forms the response, directly interacting with the client.



### 2.2. Consumer



-   **What it is:** Handles inbound asynchronous messages (e.g., from message queue). Responsible for message deserialization, invoking the corresponding `Use Case`/`Query`/`Handler`, and managing message lifecycle (ack/nack).

-   **Location (Heuristic):** Path to module contains `consumers`, `consumer`, `listeners`, `listener`, `subscribers` or `subscriber`.

-   **Naming (RegEx):** Module (file) name matches `.*consumer$`, `.*subscriber$`, `.*listener$`, `.*handler$`.

-   **Key difference:** Asynchronously processes messages, initiating processes in the application without direct response to the sender.



---



### Key difference: Adapter vs Port

-   **[[ports|Port]] (`ports/driven/db/sqlalchemy_session.py`):** This is `SQLAlchemy Session`. It can do `add()`, `commit()`, `execute("SELECT...")`. It is the **tool**.

-   **[[adapters|Adapter]] (`adapters/driven/repositories/sql_invoice_repository.py`):** This is `InvoiceRepository`. It uses the **Port** (`SQLAlchemy Session`) to persist the specific `Invoice` (Domain) to the database. It is **using the tool**.

# Anatomy: [[app|app]] (Orchestration)



The `app` layer is the control center that coordinates business scenarios. It does not contain business rules (those are in [[domain]]) and is abstracted from infrastructure details ([[adapters]]).



Components of this layer are typically located in the directory `app/` within a given [[2_MyProjects/dddguard/guidelines/context_content/context_composition|Bounded Context]] (e.g., `billing/app/` or `shipping/app/`).

    â”‚   â””â”€â”€ get_invoice_details_query.py

    â”œâ”€â”€ handlers/

```

### 1. Use Case



-   **What it is:** Implements an atomic business scenario that changes system state ([[#1-use-case|Command]]). Orchestrates interaction between [[domain|domain layer]] and [[adapters|adapters]].

-   **Location (Heuristic):** Path to module contains `use_cases`, `usecase`, `ucs`, `commands`, `interactors` or `actions`.

-   **Naming (RegEx):** Module (file) name matches `.*use_?case$`, `.*command$`, `.*action$`, `.*interactor$`.

-   **Key difference from [[#2-query|Query]]**: Changes system state and is usually transactional.



### 2. Query



-   **What it is:** Implements an atomic business scenario for retrieving data ([[#2-query|Read]]). Optimized for reading, does not change system state.

-   **Location (Heuristic):** Path to module contains `queries`, `query`.

-   **Naming (RegEx):** Module (file) name matches `.*query$`.

-   **Key difference from [[#1-use-case|Use Case]]**: Read-only data access, does not change system state.



### 3. Workflow



-   **What it is:** Orchestrates long-running, multi-step processes (sagas) that may span multiple [[#1-use-case|Use Case]]s or interact with external systems. Implements compensation logic.

-   **Location (Heuristic):** Path to module contains `workflows`, `workflow`, `sagas` or `saga`.

-   **Naming (RegEx):** Module (file) name matches `.*workflow$`, `.*saga$`, `.*process$`.

-   **Key difference from `Use Case`:** Responsible for coordinating multiple scenarios, often with external systems and failover recovery logic.



### 4. Handler



-   **What it is:** Component that reacts to domain or integration events. Triggered not directly by the user, but by consumer adapters ([[adapters#2.2.-consumer-consumer|Consumers]]).

-   **Location (Heuristic):** Path to module contains `handlers` or `handler`.

-   **Naming (RegEx):** Module (file) name matches `.*handler$`.

-   **Key difference from `Use Case`:** Invoked asynchronously in response to an event, not a direct user request.



### 5. Interface



-   **What it is:** Defines the contract (port) for dependencies that will be implemented in the [[adapters|adapters]] layer. Ensures dependency inversion.

-   **Location (Heuristic):** Path to module contains `protocols`, `interfaces`, `interface`.

-   **Naming (RegEx):** Module (file) name matches `^i[A-Z].*` (starts with 'i' and capital letter), `.*interface$`, `.*port$`, `.*protocol$`.

**Key difference from [[adapters|Implementations]]**: It is an abstraction, contains no implementation.

# DDD Application Architecture



An application built on the principles of [[architecture|Domain-Driven Design (DDD)]] and clean architecture conceptually consists of three main macro-components:



1.  **[[composition|Composition Root (Assembling the entire application)]]**

    -   The only layer that "knows" about all parts of the system. It is responsible for assembling all dependencies and starting the application.



2.  **[[context_content/01-2_MyProjects - dddguard - guidelines - context_content-summary|Bounded Contexts ( cohesive contexts)]]**

    -   Isolated modules implementing a specific part of business logic (e.g., `billing`, `shipping`). Each context has its own internal architecture ([[context_content/domain|domain]], [[context_content/app|app]], [[context_content/adapters|adapters]]).



3.  **[[shared_kernel|Shared Kernel (Common Core)]]**

    -   Special context containing shared code ([[context_content/domain#1-value-object-valueobject|Value Objects]], [[context_content/dto|DTO]], interfaces) that can be reused by other contexts.

# composition



# Anatomy: Composition Root (Assembling the entire application)



**Composition Root** is the only application-wide layer that knows about the existence of all Bounded [[context_content/01-2_MyProjects - dddguard - guidelines - context_content-summary|Contexts]], Shared Kernel ([[shared_kernel|Shared Kernel]]), and concrete implementations. Its task is to assemble all parts of the application into a single working mechanism and provide entry points for launch.



This layer is not part of any context but stands above them, orchestrating their assembly and interaction.



### Example structure



```

src/

â”œâ”€â”€ billing/                # Context 1

â”œâ”€â”€ shipping/               # Context 2

â”œâ”€â”€ shared_kernel/          # Shared Kernel

â””â”€â”€ root/                   # <-- Composition Root

    â”œâ”€â”€ root.py             # Main assembler

    â”œâ”€â”€ main.py             # Entry point for web service (FastAPI/Flask)

    â””â”€â”€ cli.py              # Entry point for console application

```



---



### 1. Main assembler (`root.py`)



-   **What it is:** Core module that imports and initializes containers or factories from each Bounded [[context_content/01-2_MyProjects - dddguard - guidelines - context_content-summary|Context]] and from `shared_kernel`. It performs the "gluing" of inter-context interaction, passing public facades of one context to another.

-   **Logic:**

    1.  Loads application configuration.

    2.  Initializes `shared_kernel`.

    3.  Initializes each context's container, passing it [[shared_kernel|shared_kernel]] and, if needed, other contexts.

    4.  Returns the fully assembled application "super-container".

-   **Key difference:** It is the "conductor" that knows all "musicians" (contexts) and hands them "scores" (dependencies).



### 2. Entry points (`main.py`, `cli.py`, `app.py`)



-   **What it is:** Final executable scripts that launch the application. They import the main assembler (`root.py`), call it, and start the corresponding [[context_content/ports#1-driving-ports-inbound-ports|Driving Port]].

-   **Types:**

    -   `main.py`/`app.py`: Initializes and starts the web server (Uvicorn, Gunicorn), passing it the assembled web application.

    -   `cli.py`: Initializes and starts the command-line parser (Typer, Click), passing it the assembled CLI commands.

-   **Key difference:** Do not contain assembly logic. Their only task is to initiate assembly via `root.py` and launch the application.

# context_composition



# Anatomy: [[context_composition|Composition]] (Context Assembly Module)



The `composition.py` module, located at the top level of the [[01-2_MyProjects - dddguard - guidelines - context_content-summary|Bounded Context]] directory (e.g., `billing/composition.py`), is the assembly instruction and entry point to this specific context. It is the only place within the context allowed to know about all its layers [[domain]], [[app]], [[adapters]] at once.



**Golden Rule:** The context must be a "black box". The global application does not know how the context is structured inside; it only receives the ready-to-use Container of this context.



### Example structure (within context)

```

â”œâ”€â”€ app/

â”œâ”€â”€ domain/

â”œâ”€â”€ adapters/

â”œâ”€â”€ ports/

â”œâ”€â”€ dtos/

â”œâ”€â”€ logger.py

```

### 1. Wiring (Assembly)



-   **What it is:** Process of connecting context components through [[3_Knowledge/Libraries/Backend/Dishka|Dependency Injection (DI)]]. This module (`composition.py`) contains a function (e.g., `wire_billing_context()`), which initializes classes and injects their dependencies ([[adapters#1-driven-adapters-outbound-adapters|Driven Adapters]], [[app#1-use-case|App UseCases]], [[adapters#2-driving-adapters-inbound-adapters|Driving Adapters]]).

-   **Key difference:** It is the "glue" that connects abstractions (`app`) with their concrete implementations (`adapters`) *within this context*. It is the only module that knows about all layers of *this context*.



### 2. Container (Context Facade)



-   **What it is:** Facade object (e.g., `BillingContainer` class) that is encapsulated and returned by the assembly function from `composition.py`. It provides the outside world (other contexts or the main [[../composition|Composition Root]]) access to the public API and context state without revealing its internal implementation.

-   **Composition:** `start()`, `stop()` methods for lifecycle management, access to public Entrypoints (e.g., `FastAPI Router`) or other public components.

-   **Key difference:** Provides a single entry point for lifecycle management and access to public components of *this context*.

# `dddguard` Architectural Standard



This document describes the high-level architecture that `dddguard` helps design, verify, and maintain. This standard aims at maximum separation of concerns and creation of clearly defined, loosely coupled modules.



## Main Components



The architecture consists of three key top-level structures:



-   `src/<context_name>/`: Isolated **Bounded Context**, implementing a specific business capability.

-   `src/shared/`: **Shared directory** for code reused across multiple contexts.

-   `src/root/`: **Composition Root**, which assembles all parts of the application into a single whole.



---



## Bounded Context (`<context_name>/`)



It is a self-contained module with its own internal layers.



### `domain/` â€” Core business logic



Pure, isolated layer depending on nothing else.

-   **`aggregates/`**: Encapsulates business rules and protects invariants.

-   **`entities/`**: Business objects with unique identity.

-   **`vo/`**: Immutable Value Objects without identity.

-   **`services/`**: Domain operations not belonging to a specific object.

-   **`events/`**: Reflects significant events that occurred in the domain.

-   **`errors/`**: Custom, domain-specific exceptions.



### `app/` â€” Application layer (orchestration)



Orchestrates domain logic to perform specific tasks.

-   **`use_cases/`**: Implements commands that change system state.

-   **`queries/`**: Implements queries for reading system state.

-   **`workflows/`**: Manages long-running processes (sagas).

-   **`handlers/`**: Reacts to domain or integration events.

-   **`interfaces/`**: Defines contracts (ports) for driven adapters.



### `adapters/` â€” Adapter layer



Translator layer between application core and the outside world.

-   **`driving/`**: Adapters that "drive" the application (e.g., controllers receiving HTTP requests).

-   **`driven/`**: Adapters "driven" by the application (e.g., repositories going to the database).



### `dto/` â€” Data Transfer Objects



Simple, logic-less structures for data transfer.

-   **`driving/`**: Inbound data structures (requests).

-   **`driven/`**: Outbound data structures (responses, events).



### `ports/` â€” Infrastructure layer



Contains code specific to particular technologies (frameworks, drivers).

-   **`driving/`**: Entry points for inbound signals (web server, CLI application).

-   **`driven/`**: Exit points for outbound signals (DB client, HTTP client).



### `composition.py` â€” Context assembly



The "glue" that connects all internal components of the context via Dependency Injection. It is the only one that knows about all layers of *this* context.



---



## Shared directory (`shared/`)



Place for code used by multiple contexts. This can be shared domain (`shared kernel`), libraries, or utilities.



**Key rule:** Code in `shared/` must be very stable, as changes to it affect all dependent contexts. Contexts may depend on `shared/`, but `shared/` cannot depend on contexts.



---



## Composition Root (`root/`)



The only place that knows about all contexts and assembles them together.

-   **`composition.py`**: Main assembler that initializes all contexts and `shared/` components, "gluing" them together.

-   **`main.py` / `cli.py`**: Entry points that launch the application (web server or CLI) using the assembler.



---



## Key principle: Ports vs. Adapters



This important distinction helps keep the architecture clean.



-   **Port** is the **tool**. It is a low-level, technological component.

    -   *Example:* `SQLAlchemy Session` in `ports/driven/db/`. It can execute `session.commit()`.

-   **Adapter** is **using the tool** for a business task.

    -   *Example:* `InvoiceRepository` in `adapters/driven/repositories/`. It uses `SQLAlchemy Session` (Port) to persist the domain object `Invoice` (Adapter).

# Anatomy: [[domain]] (Core)



The `domain` layer is the isolated core of business logic. It does not depend on external layers ([[app]], [[adapters]]) and contains only pure business rules and models.



Components of this layer are typically located in the `domain/` directory within a given [[01-2_MyProjects - dddguard - guidelines - context_content-summary|Bounded Context]] (e.g., `billing/domain/` or `shipping/domain/`).

### 1. Value Object



-   **What it is:** Immutable [[#1-value-object|Value Object]] describing a characteristic or attribute without its own identity (e.g., `Money`, `EmailAddress`). Equality is determined by field values.

-   **Location (Heuristic):** Path to module contains `value_objects`, `vo`, `types` or `enums`.

-   **Naming (RegEx):** Module (file) name matches `.*vo$`, `.*valueobject$`, `.*value$`.

-   **Key difference:** Has no unique `ID` and is `immutable`.



### 2. Entity



-   **What it is:** Mutable [[#2-entity|object]] with unique identity ([[#2-entity|ID]]) and lifecycle (e.g., `Customer`, `Product`).

-   **Location (Heuristic):** Path to module contains `entities`, `models`.

-   **Naming (RegEx):** Module (file) name matches `.*entity$`, `.*_ent$`, `.*model$`.

-   **Key difference:** Has a stable `ID` to refer to. State may change over time.



### 3. Aggregate Root



-   **What it is:** Root of a cluster of [[#2-entity|domain objects]] ([[#2-entity|Entities]] and [[#1-value-object|Value Objects]]), that ensures transactional integrity and protects business invariants. All external access to the cluster goes only through [[#3-aggregate-root|Aggregate Root]].

-   **Location (Heuristic):** Path to module contains `aggregates` or `agg`.

-   **Naming (RegEx):** Module (file) name matches `.*aggregate$`, `.*root$`.

-   **Key difference:** It is the entry point for all changes within an entire group of related objects.



### 4. Domain Service



-   **What it is:** Stateless service implementing business logic that does not naturally belong to any single [[#2-entity|Entity]] or [[#1-value-object|Value Object]]. Coordinates actions between multiple domain objects.

-   **Location (Heuristic):** Path to module contains `services`, `policies`, `logic` or `rules`.

-   **Naming (RegEx):** Module (file) name matches `.*service$`, `.*policy$`, `.*rule$`, `.*logic$`.

-   **Key difference from [[app#1-use-case|App Service]]**: Operates only with domain objects and does not interact with the outside world (DB, API). Implements **business rules**, not **execution flow**.



### 5. Domain Event



-   **What it is:** Immutable object recording the fact that an important event occurred in the domain. Used for loose coupling between system components.

-   **Location (Heuristic):** Path to module contains `events` or `domain_events`.

-   **Naming (RegEx):** Module (file) name matches `.*event$`, `.*occurred$`.

-   **Key difference from [[dto|DTO]]**: Describes an accomplished fact in the past, not a command or data for transfer.



### 6. Factory



-   **What it is:** Object encapsulating complex logic for creating domain objects ([[#2-entity|Entities]], [[#3-aggregate-root|Aggregates]]), ensuring their valid state at creation time.

-   **Location (Heuristic):** Path to module contains `factories`.

-   **Naming (RegEx):** Module (file) name matches `.*factory$`.

-   **Key difference from [[3_Knowledge/Libraries/Backend/Dishka|DI container]]**: Creates domain objects, not services or infrastructure components.



### 7. Domain Error



-   **What it is:** Specialized exception that is part of the Ubiquitous Language and describes a violation of a specific business rule.

-   **Location (Heuristic):** Path to module contains `domain_errors`, `errors` or `exceptions`.

-   **Naming (RegEx):** Module (file) name matches `.*error$`, `.*exception$`, `.*failure$`.

-   **Key difference from `Standard Exception`:** Has business meaning, not technical (e.g., `InsufficientFundsError` instead of `ValueError`).

# dto



# Anatomy: [[dto|DTO]] (Data Transfer Objects)



Data Transfer Objects are simple, logic-less objects intended for safe and structured data transfer between application layers and external boundaries. They define the public contract.



DTOs are clearly separated by data flow direction: `driving` (inbound) and `driven` (outbound). For clarity, they can be further grouped by protocol or technology.



### Example structure



```

billing/

â””â”€â”€ dto/

    â”œâ”€â”€ driving/            # Inbound DTO (Requests)

    â”‚   â”œâ”€â”€ http/

    â”‚   â”‚   â””â”€â”€ create_invoice_request_dto.py

    â”‚   â””â”€â”€ rpc/

    â”‚       â””â”€â”€ get_status_request_dto.py

    â””â”€â”€ driven/             # Outbound DTO (Responses)

        â”œâ”€â”€ kafka/

        â”‚   â””â”€â”€ invoice_finalized_event_dto.py

        â””â”€â”€ ws/

            â””â”€â”€ notification_response_dto.py

```



---



### 1. Request DTO



-   **What it is:** Inbound data transfer object used for validation and structuring of information entering *into* the application from the outside world (e.g., HTTP request). Contains no business logic.

-   **Location (Heuristic):** Module is located in directory `dto/driving/`.

-   **Naming (RegEx):** File name always contains suffix `_dto` (e.g., `create_invoice_request_dto.py`).

-   **Key difference from [[#2-response-dto|Response DTO]]:** Carries data *into* the system. May contain `to_domain()` method for conversion to [[domain|domain objects]].



### 2. Response DTO



-   **What it is:** Outbound data transfer object used for structuring and formatting information the application sends *outward* (e.g., HTTP response). Represents a projection of domain data.

-   **Location (Heuristic):** Module is located in directory `dto/driven/`.

-   **Naming (RegEx):** File name always contains suffix `_dto` (e.g., `invoice_details_response_dto.py`).

-   **Key difference from [[#1-request-dto|Request DTO]]:** Carries data *out of* the system. May contain factory method `from_domain()` for creation from [[domain|domain objects]].

## 1. Driving Ports (Inbound ports)



These ports provide mechanisms through which the outside world delivers signals *into* our application. They are implementations of low-level protocols.



-   **What it is:** Technological implementation of application entry point (e.g., web server, command-line parser).

-   **Location (Heuristic):** Path to module contains `ports/driving/`. May be subdivided by technology (e.g., `ports/driving/http/`, `ports/driving/cli/`).

-   **Naming (RegEx):** Module (file) name matches `.*server$`, `.*cli$`, `.*app$`.

-   **Key difference:** Responsible for receiving "raw" data from external source and passing it to [[adapters#2-driving-adapters-inbound-adapters|Driving Adapters]].



## 2. Driven Ports (Outbound ports)



These ports provide low-level clients and drivers that the application uses to deliver data *outward*.



-   **What it is:** Technological implementation of interaction with external systems (e.g., database driver, HTTP client).

-   **Location (Heuristic):** Path to module contains `ports/driven/`. May be subdivided by technology (e.g., `ports/driven/db/`, `ports/driven/http_clients/`).

-   **Naming (RegEx):** Module (file) name matches `.*db$`, `.*engine$`, `.*session$`, `.*client$`.

-   **Key difference:** Responsible for sending "raw" data to external source, without regard for business logic.



## 3. Bidirectional Ports (Bidirectional ports)



This category is for technologies that support a persistent two-way communication channel where either side can initiate.



-   **What it is:** Technological implementation of persistent two-way connection (e.g., WebSockets, gRPC streaming).

-   **Location (Heuristic):** Path to module contains `ports/bidirectional/`. May be subdivided by technology (e.g., `ports/bidirectional/websockets/`).

-   **Naming (RegEx):** Defined by the specific technology (e.g., `.*socket$`, `.*ws_client$`).

-   **Key difference:** Supports active data exchange in both directions, unlike unidirectional inbound/outbound requests.



---



### Key difference: Adapter vs Port

-   **[[ports|Port]] (`ports/driven/db/sqlalchemy_session.py`):** This is `SQLAlchemy Session`. It can do `add()`, `commit()`, `execute("SELECT...")`. It is the **tool**.

-   **[[adapters|Adapter]] (`adapters/driven/repositories/sql_invoice_repository.py`):** This is `InvoiceRepository`. It uses the **[[ports|Port]]** (`SQLAlchemy Session`) to persist the specific [[domain#3-aggregate-root|Invoice]] ([[domain|Domain]]) to the database. It is **using the tool**.

# Anatomy: [[shared_kernel|Shared Kernel]] (Common Core)



**Shared Kernel** is a dedicated [[2_MyProjects/dddguard/guidelines/context_content/context_composition|Bounded Context]] containing a set of shared components ([[context_content/domain#1-value-object|Value Objects]], [[context_content/dto|DTO]], [[context_content/app#5-interface|Interfaces]], utilities, base exceptions, etc.) that can be safely reused by several other [[2_MyProjects/dddguard/guidelines/context_content/context_composition|Bounded Contexts]]. Its goal is to avoid code duplication while minimizing tight coupling.



`Shared Kernel` does not have strict naming rules at the file/module level like regular contexts. However, the `dddguard` scanner still attempts to determine layers and component types within `Shared Kernel` based on its directory structure.

### Golden Rules



1.  **Dependencies:** `Shared Kernel` does not depend on other contexts. Other contexts **import** and **depend on** `Shared Kernel`.

2.  **Domain Isolation:** The [[context_content/domain|domain]] layer of any context is **forbidden** to import anything from `shared_kernel/domain`. This is necessary to preserve purity and full isolation of business logic. The [[context_content/app|app]], [[context_content/adapters|adapters]] and [[context_content/ports|ports]] layers may use `Shared Kernel`.

3.  **Stability:** Code in `Shared Kernel` should be maximally stable. Any change here potentially affects all contexts depending on it.

4.  **Composition:** Usually contains:

    *   Shared **[[context_content/domain#1-value-object|Value Objects]]** (`Money`, `EmailAddress`).

    *   Base **[[context_content/app#5-interface|interfaces/ports]]** (`IEventBus`, `ICommitter`).

    *   **[[context_content/dto|DTO]]** for inter-context exchange (`UserProfileDto`).

    *   Shared **utilities** and **helpers** (`date_helpers`).

    *   Base **exceptions** (`EntityNotFoundError`).



### Scanner classification (`dddguard`)



-   **Location (Heuristic):** Main directory `shared/` or `shared_kernel/` at the top level of the project. Inner directories (e.g., `shared/domain/`, `shared/app/`) will be classified by the scanner similarly to regular contexts.

-   **Naming (RegEx):** No strict naming rules at file/module level (no specific rules for `SHARED` in `rules_registry`). Clear, self-describing names matching their purpose are recommended.
